# Verification plan: pr-merge-policy

Every brief AC, its method, the concrete scenario that produces the evidence, and where the evidence lives. Unit/integration evidence is the named test passing in CI (`npm test`); live-e2e evidence is a validated `archie-e2e-evidence/v1` file pair; deploy-only is a named post-merge operator step.

| AC1 | integration | Mocked `GitHubClient` returns open+approved+clean for a PR in a non-auto repo; run `checkAndMergeLinkedPRs` twice back-to-back (simulating an approval/synchronize/push/workflow_run burst reaching the debounced handler more than once). Assert `mergePullRequest` is never called, the PM ready-notification (finding + `sendMessage` to `pm-agent`) fires exactly once across both runs, and `BranchState.merge_ready_notified` is set. Companion cases: marker persists across task reload (no re-notify); PR observed not-ready clears the marker and a later ready state re-notifies once; a ready PR whose `pending_merge_approval` slot matches produces no notification (approval prompt already pending — no double nudge). Note the engine-side once-ness (one finding + one `sendMessage`) is a proxy for the PM's single Slack post, which is LLM behavior — closed by the updated engineering-team skill (task 8.4) and observed live in the AC9 checklist. | `src/connectors/github/__tests__/merge.test.ts` |
